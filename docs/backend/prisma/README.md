# Prisma — schema & migrations

Design-stage database layer for the WebHack Academy backend. Mirrors
[`../01-database-schema.md`](../01-database-schema.md).

## Files
| File | What |
| --- | --- |
| `schema.prisma` | Full data model — 56 models, 21 enums, relations, cascade rules, indexes, soft-delete columns, timestamps. **Validated** with Prisma 6 (`prisma validate`). |
| `migrations/20260722000000_init/migration.sql` | Initial migration SQL, **generated by `prisma migrate diff`** (authoritative, not hand-written). 56 tables · 21 enum types · 75 indexes · 80 FK constraints · `citext` extension. |
| `migrations/migration_lock.toml` | Provider lock (postgresql). |
| `soft-delete.ts` | Prisma Client extension implementing the soft-delete strategy. |

## How this migration was generated
No database was touched. The SQL was produced offline from the schema:
```bash
npx prisma@6 migrate diff \
  --from-empty \
  --to-schema-datamodel schema.prisma \
  --script > migrations/20260722000000_init/migration.sql
```
This is exactly what `prisma migrate dev --name init` emits, minus the DB round-trip.

## Running it for real (once the backend + DB exist)
```bash
# 1. Point Prisma at a database
export DATABASE_URL="postgresql://user:pass@localhost:5432/webhack?schema=public"

# 2. Apply migrations
npx prisma migrate deploy        # prod: apply committed migrations
# or, in development:
npx prisma migrate dev           # apply + regenerate client on schema changes

# 3. Generate the typed client
npx prisma generate

# 4. Seed reference/sample data (mirror src/mocks/db.ts)
npx prisma db seed
```

## Conventions baked into the schema

### Primary keys
UUID **v7** (`@default(uuid(7))`), time-sortable — good index locality, no central sequence. Requires Prisma ≥ 5.14.

### Timestamps
Every model has `createdAt DateTime @default(now())` and (where mutable)
`updatedAt DateTime @updatedAt` — Prisma maintains `updatedAt` automatically. Columns
map to snake_case (`created_at`, `updated_at`).

### Soft delete
`deletedAt DateTime?` on **User**, **Course**, **DiscussionComment** (content that must be
recoverable / preserve referential history). Enforced by `soft-delete.ts`:
- `delete`/`deleteMany` → `update` setting `deletedAt = now()`.
- reads auto-filter `deletedAt = null`.
Hard delete is reserved for GDPR erasure and GC jobs. Indexes on `deleted_at` keep the
default filter cheap.

### Cascade rules (summary)
- **CASCADE** (child dies with parent): modules→course, lessons→module, quiz/assignment
  children, enrollment progress, refresh tokens/sessions/2FA→user, order items/payments→order,
  message/participants→conversation, etc.
- **SET NULL** (keep child, drop link): `enrollment.lastLessonId`, `course.thumbnailId`,
  `lesson.mediaAssetId/quizId`, `submission.gradedById`, `auditLog.actorId`,
  `category.parentId`, comment `parentId`.
- **RESTRICT** (block delete): course→instructor, course→category, orderItem→course,
  message→sender, subscription→plan, resource/attachment→mediaAsset — these use soft delete
  or explicit reassignment instead of cascading data loss.

### Money & JSON
Money is `Int` minor units + `Char(3)` currency. Flexible payloads (`variants`, `metadata`,
`data`, quiz `options/correct`) are `Json` (jsonb).

### Indexes
FKs, unique business keys (`email` citext, `slug`, `credential_id`, `code`), hot list/sort
paths (`enrollments(userId,status)`, `notifications(userId,createdAt)`,
`messages(conversationId,createdAt)`, `courses(status,publishedAt)`, `audit_logs(actorId,at)`),
and composite uniqueness for one-per-relationship rules.

> **Prisma 7 note:** Prisma 7 moved the datasource `url` out of `schema.prisma` into a
> `prisma.config.ts`. This schema targets **Prisma 6** (current stable). To use Prisma 7,
> remove the `url` line and add a config file per the Prisma 7 migration guide.
